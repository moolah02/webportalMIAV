1. System Overview

The Revival Technologies MIAV System (Man In A Van) is a field service management platform built to manage POS terminal deployments, technician job assignments, asset tracking, support ticketing, and client management across multiple regions.

The system consists of two primary surfaces:

  • Web Dashboard — A browser-based administrative and management portal used by administrators, managers, and supervisors.
  • Mobile Application — A smartphone application used by field technicians to receive job assignments, submit site visit reports, manage assets, and raise support tickets.

Both surfaces communicate with the same backend API, ensuring real-time data consistency across all users and roles.

4. Component Descriptions

4.1 Web Dashboard

The web dashboard is a Laravel Blade application served from the same server as the API. It provides:

  • Administrator control panel for managing employees, roles, and permissions
  • Client and POS terminal management with import/export
  • Job assignment creation and monitoring
  • Reporting and analytics dashboards
  • Asset inventory management and approval workflows
  • Ticket management and support centre
  • Document hub (this portal)

4.2 Mobile Application

The mobile app is used exclusively by field technicians. It communicates with the backend exclusively via the REST API. Key capabilities:

  • Login and secure token-based session
  • View and manage assigned jobs
  • Submit site visit reports with photos and signatures
  • Scan and update terminal status
  • Raise and update support tickets
  • Submit asset requests
  • Offline-first support — data syncs when connectivity is restored

4.3 API Layer

The RESTful API is built with Laravel and protected by Sanctum bearer tokens. All API routes are prefixed with /api. The API serves both the mobile app and any future integrations.

  • 100+ endpoints across 15+ functional areas
  • Consistent JSON response format
  • Role-scoped data access
  • Offline sync endpoints for batch data retrieval
  • Validation on all write operations

5. Data Flow

5.1 Technician Job Flow

  1. Admin creates a Job Assignment in the web dashboard and assigns it to a technician
  2. Technician receives the assignment on the mobile app
  3. Technician travels to site, checks in, and performs service
  4. Technician submits a Site Visit Report with terminal status, photos, and signature
  5. Report is synced to the server and visible in the web dashboard immediately
  6. Admin marks the job as complete or raises follow-up tickets

5.2 Ticket Flow

  1. Ticket created (by technician from mobile or admin from web)
  2. Ticket status: open → in_progress → resolved / closed
  3. Staged resolution: steps can be assigned to different employees, transferred, and completed sequentially
  4. Resolution notes recorded; client notified

5.3 Offline Sync Flow

  1. Mobile app caches assignments, terminals, and assets locally on login
  2. Technician works offline — data stored locally
  3. On reconnect, app calls /api/sync/bulk-sync to push pending records
  4. Server reconciles and returns any updates since last sync timestamp

6. Authentication & Security

6.1 API Authentication

The API uses Laravel Sanctum bearer tokens. On successful login, the server issues a plain-text token which the mobile app stores securely and includes in every subsequent request as an Authorization: Bearer {token} header.

6.2 Role-Based Access Control

RoleAccess Level
Super AdminFull system access — all modules, settings, user management
AdministratorAll operational modules — clients, terminals, jobs, tickets, assets
ManagerTeam oversight — assignments, reports, technician management
SupervisorRegional oversight — assignments and visits for assigned regions
TechnicianOwn assignments, site visits, tickets assigned to them, asset requests
Client UserRead-only view of their own terminals, tickets, and project data

6.3 Security Measures

  • All communication over HTTPS (TLS)
  • Passwords hashed with bcrypt (12 rounds)
  • AES-256-CBC encryption for sensitive data at rest
  • Token revocation on logout
  • Input validation on all write operations
  • SQL injection prevention via Eloquent ORM
  • CSRF protection on web forms

8.2 Webhook Endpoints

The system exposes webhook receivers for external system integration:

  • POST /api/webhooks/terminal-update — Accepts terminal status updates from external systems
  • POST /api/webhooks/job-completion — Accepts job completion callbacks
Future integrations (payment gateways, ERP systems, SMS notifications) can be added via the existing API layer without architectural changes.
