3. Scope of Services

This agreement covers the following services:

Included

  • Bug fixes for defects in the delivered system
  • Security patches for critical vulnerabilities
  • Database maintenance (migrations, optimisation)
  • Server deployment support (code updates, cache clearing)
  • Performance issue investigation
  • Technical support via agreed communication channels
  • Documentation updates for bug fixes and minor changes

Not Included (see Section 8)

  • New features or functionality changes
  • Third-party service issues (AWS outages, mobile OS updates)
  • Data entry errors by users
  • Mobile application development changes

7. Responsibilities

Service Provider Responsibilities

  • Respond to reported issues within agreed SLA timeframes
  • Investigate, diagnose, and resolve bugs in the delivered system
  • Deploy fixes to the production server following agreed change control process
  • Communicate progress and estimated resolution times
  • Maintain confidentiality of client data
  • Provide updated documentation for any changes made

Client Responsibilities

  • Report issues through the agreed channel with sufficient detail (steps to reproduce, screenshots, error messages)
  • Provide access to server and systems when required for investigation
  • Ensure server infrastructure (AWS EC2) remains active and paid
  • Maintain the SSH access key (RTLDEV.pem) securely
  • Ensure users are trained and do not misuse the system
  • Test and confirm fix resolution within 5 business days of deployment
  • Ensure domain name and SSL certificates remain active

8. Excluded Services

The following are not covered under this SLA and require a separate change order or agreement:

  • New features, functionality, or modules not in the original delivery
  • Changes to the mobile application code
  • Third-party service outages (AWS, GitHub, email provider, etc.)
  • Issues caused by client-side changes to the server or database without prior coordination
  • Data recovery from user errors (e.g., accidental deletion of records)
  • Training sessions for new users
  • Hardware or network infrastructure management
  • Custom integrations with external systems
  • Issues arising from use of unsupported browser versions or mobile OS versions

10. Issue Reporting Process

All issues must be reported through the agreed support channel. When reporting, include:

  1. Description: What happened vs what was expected
  2. Steps to reproduce: Exact steps taken before the issue occurred
  3. Severity: Your assessment of the impact (Critical / High / Medium / Low)
  4. Screenshots or video: Visual evidence of the issue
  5. Error messages: Exact text of any error messages shown
  6. User affected: Role and account email of the affected user
  7. Environment: Production or Development
The more detail provided, the faster the issue can be diagnosed and resolved. Incomplete reports may result in delays while additional information is gathered.

11. Escalation Process

If an issue is not being resolved within the agreed SLA timeframe:

  1. Contact the assigned support representative and request an update
  2. If no response within 2 hours of the SLA breach, escalate to the project lead
  3. If still unresolved, escalate to senior management of the service provider

Persistent SLA breaches should be documented and may be used to renegotiate the agreement or apply agreed penalties as specified in the commercial contract.
