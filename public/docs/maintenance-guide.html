2. Deploying Updates

2.1 Standard Code Deploy (Production)

Use this procedure whenever new code is pushed to the main branch on GitHub.

1
Connect to the server via SSH 
ssh -i "C:\Users\hp\Downloads\RTLDEV (1).pem" ubuntu@51.21.252.67
2
Navigate to the production directory 
cd /var/www/html/revival_production
3
Pull the latest code 
sudo -u www-data git -c safe.directory=$(pwd) pull origin main
4
Run any new database migrations 
sudo -u www-data php artisan migrate --force
5
Clear all caches 
sudo -u www-data php artisan cache:clear
sudo -u www-data php artisan config:clear
sudo -u www-data php artisan view:clear
sudo -u www-data php artisan route:clear
Before deploying to production, always deploy to the development environment first and verify everything works correctly.

2.2 Deploy to Development First

cd /var/www/html/revival_dev
sudo -u www-data git -c safe.directory=$(pwd) pull origin main
sudo -u www-data php artisan migrate --force
sudo -u www-data php artisan cache:clear

2.3 If Frontend Assets Changed (Vite)

If CSS, JavaScript, or Vue components were modified, rebuild the assets:

npm run build
Frontend asset builds must be done locally and committed. The server does not run Node.js for builds.

3. Database Backups

3.1 Manual Database Backup

Run this command on the server to create a backup of the production database:

# Replace DB_NAME, DB_USER, DB_PASS with actual values from .env
mysqldump -u DB_USER -pDB_PASS DB_NAME > /home/ubuntu/backups/miav_backup_$(date +%Y%m%d_%H%M%S).sql

# Compress the backup
gzip /home/ubuntu/backups/miav_backup_*.sql

3.2 Automated Daily Backup (Recommended Setup)

Add the following cron job to automate daily backups:

# Edit crontab: crontab -e
0 2 * * * mysqldump -u DB_USER -pDB_PASS DB_NAME | gzip > /home/ubuntu/backups/miav_$(date +\%Y\%m\%d).sql.gz
Keep at least 30 days of backups. Older backups should be archived to S3 or another off-site location.

3.3 Backup Storage Locations

AWS S3 (recommended)90 daysOff-site disaster recovery

4. Restoring from Backup

Restoration overwrites the current database. Always take a fresh backup of the current database before restoring from an old backup.
1
Take a backup of current database (safety) 
mysqldump -u DB_USER -pDB_PASS DB_NAME > /home/ubuntu/backups/pre_restore_backup.sql
2
Decompress the backup file (if gzipped) 
gunzip /path/to/backup_file.sql.gz
3
Restore the database 
mysql -u DB_USER -pDB_PASS DB_NAME < /path/to/backup_file.sql
4
Clear Laravel caches 
sudo -u www-data php artisan cache:clear
sudo -u www-data php artisan config:clear
5
Verify the application is working correctly

Test login and key features on the web dashboard and via API.

5. Monitoring & Health Checks

The API exposes a public health check endpoint. No auth required:

GET /api/test
GET /api/public/status

Expected response: {"success": true, "status": "operational"}

6. Log Management

6.1 Laravel Application Logs

# View latest errors (last 100 lines)
tail -n 100 /var/www/html/revival_production/storage/logs/laravel.log

# Watch logs in real-time
tail -f /var/www/html/revival_production/storage/logs/laravel.log

# Search for specific errors
grep "ERROR" /var/www/html/revival_production/storage/logs/laravel.log | tail -20

6.2 Nginx Access & Error Logs

# Access log (all requests)
tail -n 100 /var/log/nginx/access.log

# Error log
tail -n 100 /var/log/nginx/error.log

6.3 Log Rotation

Laravel logs rotate daily by default. Nginx uses system logrotate. Logs older than 14 days are automatically compressed.

If the storage/logs/laravel.log file grows very large, you can safely truncate it (after reviewing errors): truncate -s 0 storage/logs/laravel.log

7. Troubleshooting Common Issues

Issue: API returns 500 Server Error

  1. Check the Laravel log: tail -n 50 storage/logs/laravel.log
  2. Look for the error message and stack trace
  3. Common causes: missing .env value, failed DB connection, missing migration
  4. Try: php artisan config:clear && php artisan cache:clear

Issue: API returns 419 CSRF token mismatch

This only affects web form submissions, not API requests. Clear session: php artisan session:flush

Issue: Migrations fail on deploy

  1. Check the error output — often a missing column or duplicate migration
  2. If a migration was already partially applied: check migrations table in DB
  3. Fix the migration file and re-run, or manually apply the SQL if safe

Issue: Website shows blank page / 502 Bad Gateway

# Check Nginx is running
sudo systemctl status nginx
sudo systemctl restart nginx

# Check PHP-FPM is running
sudo systemctl status php8.2-fpm
sudo systemctl restart php8.2-fpm

# Check Nginx config syntax
sudo nginx -t

Issue: Git pull fails with "dubious ownership"

sudo -u www-data git -c safe.directory=/var/www/html/revival_production pull origin main

Issue: Application out of memory / slow

# Check memory usage
free -h

# Clear all Laravel caches
php artisan optimize:clear

# Restart PHP-FPM to free memory
sudo systemctl restart php8.2-fpm

Issue: Mobile app shows "Unable to connect to server"

  1. Verify the server is running: sudo systemctl status nginx
  2. Check the API URL the mobile app is using matches the production URL
  3. Test the health endpoint from a browser: /api/public/status
  4. Check SSL certificate is valid and not expired
  5. Check firewall rules allow traffic on port 80 and 443

Issue: Login fails for a specific user

  1. Check the user's account status in the web dashboard (Admin → Employees)
  2. Ensure account status is set to "Active"
  3. Reset their password from the admin panel if needed

8. Server Management

8.1 Restart Services

# Restart Nginx (web server)
sudo systemctl restart nginx

# Restart PHP-FPM (PHP processor)
sudo systemctl restart php8.2-fpm

# Restart MySQL
sudo systemctl restart mysql

# Restart all three
sudo systemctl restart nginx php8.2-fpm mysql

8.2 Reboot Server

Rebooting the server will briefly take the application offline (typically 1–2 minutes). Notify users before doing so.
sudo reboot

8.3 Check Server Uptime

uptime

8.4 Update Server Packages

sudo apt update && sudo apt upgrade -y
Always test on dev before updating server packages, as PHP or Nginx upgrades can break the application.

9. SSL Certificate Renewal

If using Let's Encrypt (Certbot):

# Check certificate expiry
sudo certbot certificates

# Renew certificate
sudo certbot renew

# Test auto-renewal (dry run)
sudo certbot renew --dry-run

Let's Encrypt certificates expire every 90 days. Certbot typically auto-renews via a cron job. Check if auto-renewal is configured:

sudo systemctl status certbot.timer

10. Emergency Procedures

10.1 Complete System Down

1
Check server is reachable — try to SSH in. If SSH fails, check AWS console for server status.
2
Restart all services
sudo systemctl restart nginx php8.2-fpm mysql
3
Check logs for errors
tail -n 50 /var/www/html/revival_production/storage/logs/laravel.log
tail -n 50 /var/log/nginx/error.log
4
If a bad deployment caused the issue — revert to the previous commit:
cd /var/www/html/revival_production
sudo -u www-data git -c safe.directory=$(pwd) revert HEAD
sudo -u www-data php artisan cache:clear
5
If database is corrupted — restore from the most recent backup (see Section 4).

10.2 Data Loss / Corruption

  1. Stop writing to the database immediately if possible (take the app offline)
  2. Take a current snapshot of the database (even if corrupted)
  3. Identify the most recent clean backup
  4. Restore from backup (see Section 4)
  5. Inform affected users and document what data may have been lost
Never attempt to fix data corruption by editing the database directly without a backup. Always snapshot first, then fix.
