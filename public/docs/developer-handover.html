4. Key Patterns & Conventions

4.1 API Response Pattern

All API responses return a consistent structure. Always use:

return response()->json([
    'success' => true,
    'message' => 'Optional message',
    'data'    => $payload,
], 200);

4.2 Validation Pattern

Use Laravel's Validator::make() for API endpoints (not $request->validate()) so you can return JSON errors:

$validator = Validator::make($request->all(), [
    'field' => 'required|string|max:255',
]);
if ($validator->fails()) {
    return response()->json([
        'success' => false,
        'message' => 'Validation failed',
        'errors'  => $validator->errors(),
    ], 422);
}

4.3 Soft Deletes

Most models do NOT use soft deletes. Hard deletes are used. Check each model individually before adding delete logic.

4.4 Transactions

Use DB::transaction() for any operation that writes to multiple tables (e.g., visit + visit_terminals).

4.5 JSON Columns

Several models use JSON columns (e.g., terminal, evidence, pos_terminals on assignments). These are cast in the model:

protected $casts = [
    'terminal'   => 'array',
    'evidence'   => 'array',
];

5. Authentication System

The system uses Laravel Sanctum for API token authentication. The Employee model (not the default User model) is the authenticatable model.

Login Flow

  1. POST /api/auth/login with email + password
  2. Server calls Auth::attempt() against the employees table
  3. Checks $employee->isActive() — returns 403 if inactive
  4. Creates a Sanctum token: $employee->createToken('mobile-app-token')->plainTextToken
  5. Returns token to client. Client stores it and sends as Authorization: Bearer {token}

Logout

Calls $request->user()->currentAccessToken()->delete() — revokes the token from the DB.

The auth guard for API routes is auth:sanctum. The web dashboard uses session-based auth. Do not mix these guards.

6. Role & Permissions System

The system uses a custom permissions system alongside Spatie Laravel Permission. Permissions are stored in both a permissions table and as JSON on roles.

Checking Permissions

// Via custom method on Employee model:
$user->hasPermission('manage_assets')

// Via Spatie can() helper:
$user->can('manage_team')

Role Hierarchy

Role SlugAccess Level
super_admin / adminAll permissions
managerTeam management, reports, assignments
supervisorRegional oversight
field_technicianOwn data only — jobs, visits, tickets, assets
client_userRead-only view of their own client data

Role Detection Methods on Employee Model

$user->isFieldTechnician()   // checks role slug
$user->isClientUser()
$user->isAdmin()
When adding new API endpoints that should be role-restricted, always check $user->isFieldTechnician() and scope data accordingly.

7. API Design Patterns

7.1 Route Registration

All API routes are in routes/api.php. Protected routes are wrapped in:

Route::middleware('auth:sanctum')->group(function () {
    // protected routes here
});

7.2 Scoped Data (Role-Based)

Several controllers scope data by role. Pattern used in TicketController@index:

if ($user->isFieldTechnician()) {
    $query->where('assigned_to', $user->id)
          ->orWhere('technician_id', $user->id);
} elseif ($user->isClientUser()) {
    $query->where('client_id', $user->client_id);
}
// else: admin/staff sees all

7.3 Schema::hasColumn Guards

Some code uses Schema::hasColumn() guards for safety during migrations. This is intentional — do not remove them without verifying the column exists in all environments.

7.4 Offline Sync Endpoints

The /api/sync/* routes return large payloads intended for local caching on the mobile app. These are intentionally unpaginated. Do not add pagination to sync endpoints.

8. Database Conventions

  • All tables use snake_case naming
  • Primary keys: id (bigInteger, auto-increment)
  • Foreign keys follow: {table_singular}_id pattern
  • All tables have created_at and updated_at timestamps
  • Enum columns are used for fixed status values — add new values via migration (ALTER TABLE)
  • JSON columns store arrays/objects — always cast them in the model
  • Auto-generated IDs (ticket_id: TCK-YYYY-XXXXXX, assignment_id: ASN-YYYYMMDD-001) are generated in controller/model, not DB

Adding New Enum Values

MySQL enum columns require a raw SQL migration to add new values:

DB::statement("ALTER TABLE tickets MODIFY COLUMN status
  ENUM('open','in_progress','pending','resolved','closed','cancelled','new_value')
  NOT NULL DEFAULT 'open'");

9. Offline Sync System

The mobile app supports offline-first operation. The sync system works as follows:

  1. On app startup, mobile calls GET /api/sync/full-sync to download all data
  2. User works offline — records are queued locally
  3. On reconnect, mobile pushes queued records to POST /api/sync/bulk-sync
  4. Server processes records and returns server-side updates since last sync
  5. The GET /api/sync/timestamp endpoint returns server time for delta comparison
Offline sync can create duplicate records if the mobile app retries failed syncs. The offline_sync_id column on the tickets table is used for deduplication — check for this before inserting.

12. Known Quirks & Gotchas

  • Employee model, not User: The authenticatable model is App\Models\Employee. The User model exists but is not used for auth.
  • assignment_id on visits is a BigInteger, not a string: The mobile app sends numeric IDs. The old validation required a string — this was fixed. Do not re-add the 'string' rule.
  • Ticket status enum: The original migration had only 4 statuses. A migration was added to include cancelled and pending. Always run migrations after pulling.
  • Public tickets: Tickets with assignment_type = 'public' can be updated by any field technician. Direct tickets are restricted to the assigned technician.
  • Asset date validation: needed_by_date uses after_or_equal:today — same-day requests are allowed.
  • Git safe.directory: The web server runs as www-data but the git repo may be owned differently. Use git -c safe.directory=/path pull to bypass the ownership check.
  • Two migration files create visits table: 2025_11_11_081245 and 2026_01_09_000003 both create a visits table. The newer one is the canonical one. The older migration may conflict if migrations are reset — be careful with migrate:fresh.
  • Schema::hasColumn() checks: Several migration files and controller methods guard with Schema::hasColumn(). This is intentional for idempotent migrations — do not remove these guards.
Never run php artisan migrate:fresh on production. It drops all tables and data. Always use php artisan migrate.
